> 상태: 설계 확정, 구현 대기
> 선행 메모리: `project_design_system_pivot`, `project_readability_failure`

## 1. 문제 / 전략

### Problem
실제 배포 결과 독자가 "안 읽힌다"고 평가했다([[project-readability-failure]]).
3대 실패:
1. **불일치** — "서로 다른 카드뉴스인 줄 알았다" (검정→흰→갈색 배경이 한 시리즈 안에서 뒤섞임)
2. **흰 박스 이질감** — 어두운 카드 위에 흰 박스를 얹어 디자인이 끊김
3. **밀도/가독성** — 작은 가운데 글씨, 위계 없음

근본 원인은 **코드 구조**다. `CoverCard`·`Compare2Card` 등 12개 템플릿이 각자
자기 배경·색·폰트·박스를 직접 정의하는 **"섬(island)"** 이라 공유 피부 층이 없다.
예: `Compare2Card`는 `background:'rgba(255,255,255,0.96)'`(흰박스), `fontSize:60`,
`#fff`/`#212529`(잉크), `borderRadius:22`를 직접 박는다. 피부에서 받는 토큰은
`--theme-primary`/`--theme-dark` 둘뿐. 그래서 템플릿마다 룩이 달라진다.

### Decision
카드 렌더링을 **피부(skin) / 뼈대(skeleton) 분리** 시스템으로 재설계한다.
- **피부 = 고정**: 토큰(`--set-*`) + 공유 컴포넌트 라이브러리. 세트당 한 번 정의.
- **뼈대 = 변함**: 배치(layout)만 담당. 자기 색·배경·폰트·박스를 정의 못 함.
- **강제 규칙**: 뼈대는 피부가 준 컴포넌트만 조립한다 → 배치가 달라도 한 시리즈 보장.

첫 세트(딥틸) 하나를 끝까지 완성하고, 기존 섬 12개를 철거한다.

### Rationale
- 불일치·흰박스 실패는 "섬 구조"의 직접 증상 → 구조를 고치면 증상이 사라진다.
- 이 시스템이 곧 북극성(템플릿-세트 DB)의 첫 세트 씨앗 — 새 세트 = 토큰값만 교체.
- moat는 부품이 아니라 **파이프라인**(논문→검증된 내용→적응형 배치→읽히는 카드).
  에디터·렌더는 commodity이므로 손코딩 최소화, 로직에 집중.

### 비-moat / moat 경계
```
commodity (다른 곳과 다르지 않게):  에디터 UX · 컴포넌트 모양 · PNG 렌더
moat (우리가 만드는 것):           논문 → grounding 검증 → 적응형 배치 → 읽히는 카드 + 편집가능
```

## 2. 아키텍처

### 2.1 피부 / 뼈대 분리

```
CardFrame (토큰 주입 지점)
  └─ injects --set-* tokens onto scope
      └─ Skeleton (배치 함수: 8종)
          └─ composes only Skin Components (18종)
              └─ Skin Components read --set-* tokens
```

- **피부(SET/THEME)** = `--set-*` 토큰 + 18개 공유 컴포넌트. 세트당 한 번 정의.
- **뼈대(SKELETON)** = 8종. 콘텐츠 배열을 받아 컴포넌트를 배치하는 **함수**.
- **🔒 강제 규칙 (핵심 불변식)**:
  - 뼈대는 색/배경/폰트/박스 스타일 prop을 **받지도 정의하지도 않는다**.
  - 스타일을 가진 primitive는 **18개 컴포넌트뿐**.
  - 새 세트 = `--set-*` 토큰값만 교체. 뼈대·컴포넌트 코드는 안 건드림.
  - 이 규칙이 "한 시리즈로 보임"을 코드로 보장한다.

### 2.2 토큰 주입 지점 — CardFrame 확장
현재 `CardFrame`은 `--theme-primary`/`--theme-dark`/`--theme-bg` 3개만 주입한다.
이를 **세트 객체 → `--set-*` 전체 토큰 주입**으로 확장한다. 앱-크롬 OKLCH 토큰
(`globals.css :root`)과 **분리**된 별도 네임스페이스(`--set-*`)를 쓴다.
주입 위치는 단일(`CardFrame` inner div의 inline style) — 누출 없음.

### 2.3 렌더 파이프라인 (변경 없음)
React 컴포넌트(skin/skeleton) → Playwright `goto` 스크린샷 → PNG (기존 S7,
[[project-s7-react-migration]]). 새 의존성 0, 비용 0. 캔버스 라이브러리(Konva/Fabric)
불필요 — "제약 편집" 모델이라 자유 캔버스가 필요 없다.

## 3. 피부층 — 토큰

카드 전용. 앱-크롬 토큰과 **분리**(`--set-*` 네임스페이스). CardFrame에서 주입.

### 3.1 색 — 3색 규율 (배경/강조/잉크)
```
--set-bg            카드 베이스 배경 (그라데이션 시작색)
--set-bg-grad       배경 그라데이션 끝색
--set-accent        강조색 (핵심어·수치·막대)
--set-accent-ink    강조색 위에 올라가는 텍스트색
--set-ink-strong    본문 최강 잉크 (제목)
--set-ink-muted     2차 잉크 (본문)
--set-ink-faint     3차 잉크 (캡션·eyebrow)
--set-surface       반투명 표면 (TintedBox 등)
--set-surface-border  표면 테두리
```
**3색 규율**: 한 세트는 배경 톤 1 + 강조 1 + 잉크 스케일. 흰박스 금지 —
표면은 `--set-surface`(반투명 틴트)로만. 이것이 흰박스 이질감 실패의 직접 수정.

### 3.2 타이포 — 1080 절대 px (위계 고정)
```
--set-display   88   BigStat 수치
--set-headline  60   카드 제목 (핵심어 em 강조)
--set-subhead   32   부제
--set-body      26   본문
--set-caption   19   캡션·출처
--set-eyebrow   18   상단 라벨 (letter-spacing 넓게)
```
폰트: Pretendard (woff2). 핵심어 강조는 `<em>` → `--set-accent` 색 + weight.

### 3.3 간격 / 모양
```
--set-pad          64   카드 안쪽 여백
--set-gap          24   요소 간 기본 간격
--set-maxw         960  콘텐츠 최대폭
--set-radius-box   22   박스 모서리
--set-radius-pill  100  pill 모서리
```

## 4. 피부층 — 컴포넌트 18개

8개 뼈대가 실제로 요구하는 것에서 도출(orphan 0). "풍부함"은 개수가 아니라
조합·적응·피부변주·craft에서 나온다(뒤집힌 U자 곡선). 컴포넌트는 **내용이 필요로
할 때만** 추가한다.

### 텍스트 (5)
| 컴포넌트 | 역할 |
|---|---|
| `Eyebrow` | 상단 카테고리 라벨 (eyebrow px, faint, 자간 넓게) |
| `Headline` | 카드 제목, 핵심어 `<em>` → accent 강조 |
| `Subhead` | 부제 (subhead px) |
| `Body` | 본문 (body px, `word-break:keep-all`) |
| `Caption` | 캡션 (caption px, faint) |

### 컨테이너 / 마크 (6)
| 컴포넌트 | 역할 |
|---|---|
| `Pill` | 라벨 알약 (accent 배경 + accent-ink) |
| `TintedBox` | 반투명 표면 박스 (`--set-surface`, **흰박스 대체**) |
| `BrandMark` | POLYINSIGHT 로고 마크 |
| `BgMotif` | 배경 모티프 (동심원·광원 등 faint 장식) |
| `AccentDivider` | 강조 구분선 |
| `IconChip` | 아이콘 칩 (Grid 항목용) |

### 복합 / 특수 (6)
| 컴포넌트 | 역할 |
|---|---|
| `SourceTag` ★ | 출처칩 (📄 출처: paper · section) — **우리만 필요** |
| `StepFlow` | 단계 흐름 (1→2→3, N단계 적응) |
| `BigStat` ★ | 큰 수치 + 단위 (display px) — **우리만 필요** |
| `CompareBars` | 비교 막대 (배열 N개 적응: 2·3·4) |
| `LabeledRow` | 라벨+값 행 |
| `FeatureColumn` | 세로 근거 열 (Reasons용) |

### 이미지 (1)
| 컴포넌트 | 역할 |
|---|---|
| `VisualZone` ★ | 디자인된 이미지 존 (focal point 점 포함) — **우리만 필요** |

★ = `SourceTag`·`BigStat`·`VisualZone` = 경쟁 템플릿엔 없는 fidelity 정체성의 증거.
`SourceTag`의 특별함은 칩 모양이 아니라 **S6 grounding 검증과의 결합**이다.

## 5. 뼈대 8개 + 서사 아키타입

### 5.1 실험형 서사 골격 (LOCKED, 9 beat)
KITECH 핵심 유형. 실독 논문 10편에서 도출.
```
1. 표지            [필수]
2. 훅-실생활문제    [필수]
3. 기존 한계        [선택]
4. 왜 이 소재        [선택]
5. ★ 혁신 소개       [필수]
6. ★ 제작방법 3스텝  [선택]
7. ★ 핵심성능+vs기존+출처 [필수, 병합]
8. 응용·의미        [필수]
9. 마무리/협력       [필수]
```
- 5장 압축 = 1·2·5·7·9
- 7장 표준 = + 3·6·8
- 상한 7장 (S6 출력한계 [[project-s6-output-limit]])

### 5.2 8 뼈대 (LOCKED) — beat를 생김새로 압축
| # | 뼈대 | 담는 beat | 핵심 컴포넌트 |
|---|---|---|---|
| ① | `Cover` | 1 표지 | Headline·Eyebrow·BrandMark·VisualZone·BgMotif |
| ② | `Statement` | 2 훅 · 3 한계 | Headline(큰 질문)·Body·BgMotif |
| ③ | `Feature` | 5 혁신 | Eyebrow·Headline·Body·VisualZone·AccentDivider |
| ④ | `Process` | 6 제작 | StepFlow(N단계)·Caption |
| ⑤ | `BigStatCompare` | 7 성능·vs | BigStat·CompareBars·SourceTag·Headline |
| ⑥ | `Reasons` | 4 왜 이 소재 | FeatureColumn(N개 세로근거)·Headline |
| ⑦ | `Grid` | 8 응용 | IconChip·Body (N개 격자) |
| ⑧ | `Closing` | 9 마무리 | Headline·BrandMark·SourceTag |

방법론형·리뷰형 논문도 이 8개를 재사용한다(아키타입만 다름, 세부는 후속).

### 5.3 적응형 원칙 — "템플릿 = 문법, 내용 = 문장"
뼈대는 **고정 슬롯이 아니라 배열을 받아 N개를 배치하는 함수**다.
`⑤CompareBars`·`⑥Reasons`·`⑦Grid`·`④StepFlow`는 N=2·3·4를 모두 받는다.
3번째 항목을 버리지 않는다(fidelity). 그래도 색·간격·위계는 피부가 정함(일관).
→ "템플릿이 내용에 맞춤" vs "내용을 템플릿에 박음"의 양자택일을 해소.

## 6. 데이터 모델 변경

### 현재
```ts
interface Card { card_num; template_type: string; image_url?; fields? }
interface CardTheme { primary; dark }
```

### 변경
```ts
interface Card {
  card_num: number
  skeleton: SkeletonType      // 8종 (구 template_type 대체)
  items?: CardItem[]          // 적응형 배열 (CompareBars/Reasons/Grid/StepFlow)
  fields?: Record<string, FieldValue>
  image_url?: string
  focal?: { x: number; y: number }   // VisualZone focal point (0~1)
}

interface CardSet {            // = 피부(세트). 새 세트 = seed만 교체
  set_key: string
  seed: string                 // Material 3 씨앗 (예: #0E5E60)
  tokens: SetTokens            // --set-* 전체 (seed에서 생성)
}
```
- `template_type`(12 섬) → `skeleton`(8종)으로 이전.
- 적응형 배열은 `items[]`로 표준화 (현재 `points_a`/`points_b` 같은 ad-hoc 필드 제거).
- `CardTheme{primary,dark}` → `CardSet`(전체 토큰)으로 확장. CardFrame이 주입.

## 7. 첫 세트 — 딥틸 (1차 확정안)

Material 3 씨앗 1개 → 세트 전체 팔레트 생성.
```
seed            #0E5E60  (deep teal)
--set-bg        #0E5E60 → --set-bg-grad #062E30 (155deg 그라데이션)
--set-accent    #FFC94A  (gold)
--set-accent-ink #062E30
--set-ink-strong #FFFFFF
--set-ink-muted  rgba(255,255,255,.66)
--set-ink-faint  rgba(255,255,255,.45)
--set-surface    rgba(255,255,255,.10)
--set-surface-border rgba(255,255,255,.16)
```
검증 카드: ⑤BigStatCompare (셀룰로스-COF 미세구슬 238 MPa vs PP 199 vs 무보강 142).
큰 수치 + 맥락 + 비교막대 + SourceTag + 일관 팔레트. 흰박스 0개.
씨앗만 바꾸면 다른 세트(= 북극성 DB의 단위).

## 9. 마이그레이션 순서

```
1. 토큰      --set-* 토큰 정의 + 딥틸 세트값 + CardFrame 주입 확장
2. 컴포넌트  18개 중 ⑤카드가 쓰는 6~7개 먼저 (Eyebrow·Headline·BigStat·
            CompareBars·SourceTag·BrandMark·BgMotif) → 나머지 점증
3. 뼈대      ⑤BigStatCompare 1개 먼저 완성 (목업 → 실제 React)
4. 검증      브라우저 시각 검증 (CLAUDE.md §13 — "빌드 성공"이 완료기준 아님)
5. 확장      나머지 7개 뼈대 조립 (기존 컴포넌트 80% 재사용 + 1~2개 신규)
6. 데이터    Card 모델 변경 (skeleton/items/focal) + S6 출력 매핑
7. 철거      섬 12개(CoverCard~ClosingCard) 제거, CARD_COMPONENTS 교체
8. S7 동기   render 라우트 skeleton dispatch로 교체, PNG 일치 확인
```
원칙: 추상 목록을 한꺼번에 설계하지 않는다. **카드 한 장을 끝까지** 만들고,
거기서 컴포넌트가 떨어져 나오게 한다. 18개는 5장 완성 후 자연히 쌓인 결과다.

## 10. 리스크 / 트레이드오프

| 리스크 | 대응 |
|---|---|
| 토큰/컴포넌트 동시 설계 → 인지 과부하 | 카드 1장씩 bottom-up. 한 번에 다 안 만든다 |
| 적응형 N배치가 Playwright PNG에서 깨짐 | ⑤카드 N=2·3·4 시각 검증을 게이트로 |
| 섬 철거 중 기존 데이터(`points_a` 등) 호환 | 데이터 마이그레이션을 7단계로 분리, S6 출력과 동기 |
| Pretendard 한글이 Playwright 렌더에서 누락 | 토큰 단계에서 woff2 등록 실측 |
| `--set-*` ↔ 앱크롬 `:root` 토큰 혼선 | 네임스페이스 분리(`--set-*`), 주입은 CardFrame 단일 지점 |
| "빌드 성공 = 완료" 착각 | 완료기준 = 브라우저 시각 일치 (CLAUDE.md §13) |
