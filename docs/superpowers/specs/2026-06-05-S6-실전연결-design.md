## 1. 문제 / 결정 / 근거

### Problem
디자인 시스템(8뼈대)은 완성됐으나 **S6가 여전히 구 12 섬 템플릿**(cover·hook·problem·circle3·compare2·grid4·definition·flow·data·showcase·closing·brand)과 **구 필드명**을 내놓는다. 진짜 논문을 업로드하면 신규 8뼈대가 아니라 구 템플릿으로 떨어진다 → 디자인 시스템이 파이프라인과 단절돼 있다.

### Decision
S6를 **신규 8뼈대 + 실험형 서사 아크(LOCKED 9beat)** 로 재작성한다.
- 데이터 모델·프론트 **변경 없음**(`CardSlot.fields`는 자유 딕셔너리, 프론트 `_v2` 키 등록 완료).
- S6의 **콘텐츠 계약만** 교체: `_TEMPLATE_SPEC`(8뼈대 필드 포맷) + `_SEQUENCING_RULES`(아크) + `_SYSTEM`(필드 포맷 규칙) + `_MOCK_*`(8뼈대 mock).
- **실험형만** 지원(방법론·리뷰 아키타입 후속). **DEV_MOCK_LLM 먼저** 검증.
- grounding 규율(§3) 보존: 필드마다 FieldValue(source·match_quality·risk), 정량 실패 → CRITICAL.

### Rationale
- 모델·프론트 불변이라 변경면이 S6 한 파일로 국소화(CLAUDE.md §5 계약 규율·local fix).
- 아크는 이미 LOCKED(스펙 §5.1) → S6는 자유 메뉴 선택이 아니라 아크를 **따른다**.
- KITECH 핵심이 실험형이라 실험형만으로 실전 가치 확보.

### Risks
- **필드 포맷 결합**: 프론트 파서가 정확한 구분자를 기대 — bars=`label:value:isPrimary`(`|`·`:`), reasons/items=`title:body`/`label:sub`(`|`·`:`), steps=`a|b|c`(`|`), 강조=`*핵심어*`. 프롬프트가 이 포맷을 정확히 가르쳐야 하며 어긋나면 렌더가 깨진다. → 검증 게이트에서 시각 확인.
- **정량 grounding 입도 저하(기존과 동일)**: bars/steps/items처럼 여러 사실을 한 문자열에 담는 필드는 FieldValue 1개(source 1개)만 갖는다. 구 S6도 동일(points_a·bars·steps_text). 막대별 개별 grounding은 후속.
- **테마-세트 단절**: S6의 `recommended_theme`(CardTheme primary/dark)는 신규 뼈대가 안 읽는다(뼈대는 `--set-*` 딥틸 고정). 단일 세트라 B-3에선 무의미 → 다중 세트 매핑은 후속. recommended_theme는 계속 산출하되 신규 뼈대 렌더엔 영향 없음(명시).
- **Reasons 미사용 가능**: 7장 이내 실험형 우선순위에서 process·grid가 reasons보다 우선 → Reasons(⑥)는 자동 출력엔 안 뽑힐 수 있음(에디터 수동 전환·후속 아키타입용으로 보존). §3 검토 포인트.

## 2. 뼈대 → 필드 계약 (B-3의 핵심)

구현된 8뼈대가 읽는 실제 필드명·포맷(권위 출처 = `web/src/components/cards/skeletons/*`). S6는 정확히 이 키·포맷을 내야 한다. 모든 값은 FieldValue. `headline`은 `*핵심어*`로 강조 표시(렌더 시 accent em).

| 뼈대(template_type) | beat | 필드 |
|---|---|---|
| `cover_v2` | 1 표지 | `eyebrow`(시리즈/권호), `headline`(제목 `*강조*`), `subtitle`(부제), `org`(기관) · image_url 선택 |
| `statement` | 2 훅 / 3 한계 | `eyebrow`, `headline`(큰 질문 `*강조*`), `body`(2~3문장) |
| `feature` | 5 혁신 | `eyebrow`, `headline`(`*강조*`), `body` · image_url 선택 |
| `process_v2` | 6 제작 | `eyebrow`, `headline`, `steps`(`"단계1|단계2|단계3"`, 3~5), `caption`(보조 한 줄) |
| `bigstat_compare` | 7 성능·vs·출처 | `eyebrow`, `headline`(`*강조*`), `stat_value`(숫자), `stat_unit`(단위), `stat_caption`(맥락), `bars`(`"라벨:값:1|라벨:값:0"` — `1`=우리/강조행), `source_ref`(출처) |
| `reasons` | 4 왜 이 소재 | `eyebrow`, `headline`(`*강조*`), `reasons`(`"제목:본문|제목:본문"`, 2~4) |
| `grid_v2` | 8 응용 | `eyebrow`, `headline`(`*강조*`), `items`(`"라벨:서브|라벨:서브"`, 2~4), `body`(선택 요약) |
| `closing_v2` | 9 마무리 | `eyebrow`, `headline`(`*강조*`), `body`, `source_ref`(선택) |

구분자 규칙(엄수):
- 항목 구분 = `|`, 쌍 구분 = `:`. 라벨/값 내부에 `|`·`:` 금지(프론트가 split).
- `bars`의 3번째 토큰은 `1`(강조/우리 방식) 또는 `0`. 보통 최댓값 행을 `1`.
- `headline` 강조는 `*…*` 1곳 권장(여러 곳 허용).

## 3. 실험형 아크 → card_count 매핑

`card_count`는 3~7(jobs.py `ge=3,le=7`). 9beat 중 우선순위로 N개 선택 후 beat 순서대로 배열.
출력 상한 7장(S6 출력한계, [[project-s6-output-limit]]).

beat→뼈대: 1→cover_v2, 2→statement, 3→statement, 4→reasons, 5→feature, 6→process_v2, 7→bigstat_compare, 8→grid_v2, 9→closing_v2.

우선순위(필수→선택): **1, 9, 7, 5, 2, 8, 6, 4, 3**
(표지·마무리·핵심성능·혁신은 항상; 그다음 훅·응용·제작·왜·한계)

| card_count | 선택 beat(정렬) | 뼈대 시퀀스 |
|---|---|---|
| 3 | 1·7·9 | cover_v2 · bigstat_compare · closing_v2 |
| 4 | 1·5·7·9 | + feature |
| 5 | 1·2·5·7·9 | cover · statement · feature · bigstat · closing (스펙 §5.1 5장 압축과 정확히 일치) |
| 6 | 1·2·5·7·8·9 | + grid_v2 |
| 7 | 1·2·5·6·7·8·9 | cover · statement · feature · process · bigstat · grid · closing |

> 검토 포인트: 이 우선순위에서 `reasons`(beat4)는 7장 이내에 자동 선택되지 않음. 의도(process·grid 우선)지만, "왜 이 소재"를 더 중시하려면 우선순위에서 4를 6 앞으로 올린다. 사용자 결정 필요.

## 4. S6 변경 범위 (한 파일: `backend/agents/s6_card_json.py`)

| 구성 | 변경 |
|---|---|
| `_TEMPLATE_SPEC` | 구 12개 → 신규 8뼈대 필드 포맷(§2)으로 전면 교체 |
| `_SEQUENCING_RULES` | 자유 메뉴 → 실험형 아크 + card_count 매핑(§3) 명시 |
| `_SYSTEM` | 필드 포맷 규칙(구분자·`*강조*`) 추가. grounding·가독성 원칙(A~D)·recommended_theme 블록은 유지 |
| `_USER` | template_spec/sequencing 주입부 유지(내용만 교체) |
| `_MOCK_TEMPLATES`·`_MOCK_FIELDS`·`_build_mock_card_data` | 8뼈대 mock으로 교체. card_count→§3 시퀀스. 실제 KITECH 논문(셀룰로스-COF 등) 기반 |
| `_build_card_editor_data`·`_post_process`·`_count_risk`·`_iter_field_values` | **불변**(필드명 무관, 제네릭) |

데이터 모델(`models.py`)·프론트·라우터·오케스트레이터 **불변**.

## 5. 검증 (DEV_MOCK_LLM 먼저)

1. `DEV_MOCK_LLM=True`로 mock 경로가 신규 8뼈대 cardData를 생성하는지 단위/통합 확인.
2. 그 cardData가 에디터/렌더 라우트에서 신규 뼈대로 렌더되는지(`/editor/{job}`·`/render/{job}/{n}`) — Playwright 캡처 시각 검증(슬라이스1·2와 동일 게이트).
3. card_count=3·5·7 각각 시퀀스가 §3대로 나오는지.
4. (후속) 실제 LLM 호출로 grounding 검증 — B-3 비목표.

기존 백엔드 테스트(`backend/tests/`)가 구 템플릿을 가정하면 신규로 갱신(스테이지 계약 변경 반영).

## 6. 비목표 (후속)
