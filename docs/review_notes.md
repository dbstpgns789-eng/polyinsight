# 검토 사안 (미결)
> 최종 업데이트: 2026-05-07

---

## S1 / S2 관련

### [검토 1] S2를 S1으로 흡수
- **내용**: pymupdf4llm이 이미 헤딩 구조를 만들어주므로 S2의 regex 섹션 분리는 S1의 마무리 작업과 다름없다. S2를 독립 스테이지로 유지할 이유가 약함.
- **제안**: S1 책임을 `PDF → { raw_text, page_map, section_map }` 으로 확장, S2 제거.
- **결정**: 미결

### [검토 2] PAGE 마커 삽입 (우선순위 높음)
- **내용**: 현재 설계에서 S6 LLM은 `source: { section, page }` 의 page 번호를 추론해야 한다. 이게 hallucination 위험 지점.
- **제안**: S1 추출 시 텍스트 안에 `<!-- PAGE N -->` 마커 삽입 → S6가 사실 참조로 page 번호 추출 가능.
- **결정**: 미결

### [검토 3] Markdown 표 → 자연어 변환
- **내용**: LLM은 Markdown 표보다 자연어 서술에서 수치를 더 정확하게 추출한다는 연구 결과 있음. 표 형식은 토큰 낭비 가능성.
- **제안**: pymupdf4llm이 감지한 표를 자연어 서술로 변환하는 후처리 추가.
- **결정**: 미결 (검증 후 결정 권고)

### [검토 4] 추출 전략 변경
- **내용**: 기존 계획(pdfplumber → PyMuPDF fallback)에서 pymupdf4llm을 1차로 사용하는 것으로 변경.
- **제안**: `pymupdf4llm → pdfplumber fallback` (PyMuPDF는 pymupdf4llm 내장이므로 별도 불필요)
- **결정**: 채택 확정

## 미결 항목 요약

| 항목 | 우선순위 | 내용 |
|---|---|---|
| PAGE 마커 삽입 | 높음 | S6 grounding 정확도 직접 영향 |
| S2 → S1 흡수 | 중간 | 설계 단순화, 스테이지 제거 |
| 표 자연어 변환 | 낮음 | 검증 필요 후 결정 |
